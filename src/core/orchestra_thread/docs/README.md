# OrchestraThreads Core

`OrchestraThreads` here is a standalone thread service plus a manual CLI agent interface.

This repo is intentionally built as a human-observable sandbox:

- the service persists `threads` and `thread_events` in Postgres;
- agents register callback URLs through `/agents/register`;
- delivery happens over HTTP callbacks with retry/backoff;
- inactivity wakeups are generated by the service;
- instead of LLM runtimes, you run manual CLI agents in separate terminals and send messages/statuses yourself.

## What the MVP includes

- `thread_id` as the durable identity of work
- root and child pairwise threads
- statuses: `open`, `in_progress`, `review`, `done`, `closed`
- owner rules:
  - owner can send `in_progress`, `done`, `closed`
  - peer can send `in_progress`, `review`
- `message`, `notification`, `inactive` events
- callback delivery with retry
- cascade close of child threads when a parent becomes terminal
- agent registration and heartbeat

## What the MVP does not include

- ACL and agent manifests
- parallel runtimes inside an agent
- `clear_context`
- `last_status`
- exact-once delivery

## Install

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

## Database

The service now requires Postgres.

Default connection settings:

- `ORCHESTRA_THREADS_DATABASE_URL=postgresql://orchestra:orchestra@127.0.0.1:5432/orchestra_threads`
- `ORCHESTRA_THREADS_DB_SCHEMA=public`
- `ORCHESTRA_THREADS_DB_MIN_POOL_SIZE=5`
- `ORCHESTRA_THREADS_DB_MAX_POOL_SIZE=20`

The service bootstraps its schema automatically on startup.

## Run the service

```bash
export ORCHESTRA_THREADS_DATABASE_URL=postgresql://orchestra:orchestra@127.0.0.1:5432/orchestra_threads
PYTHONPATH=src python -m core.orchestra_thread.service_main
```

The service listens on `http://127.0.0.1:8788` by default.

It also serves a read-only thread console:

- `http://127.0.0.1:8788/`
- `http://127.0.0.1:8788/ui`

The UI is backed by the same `/api/v1/threads` and `/api/v1/threads/{thread_id}` endpoints used by the CLI and tests.

## Run manual agents

Open two more terminals.

Terminal 2:

```bash
PYTHONPATH=src python -m core.orchestra_thread.agent_cli --slug secretary --listen-port 9001
```

Terminal 3:

```bash
PYTHONPATH=src python -m core.orchestra_thread.agent_cli --slug orchestra --listen-port 9002
```

For a human-driven terminal you can now skip manual port selection and start a chat-oriented CLI:

```bash
PYTHONPATH=src python -m core.orchestra_thread.agent_cli --slug human --target orchestra
```

With `--target` set, plain text lines are sent directly to that agent. You can also route one-off messages with `@agent_slug message`.

If the service runs inside Docker, advertise a host reachable from the container:

```bash
PYTHONPATH=src python -m core.orchestra_thread.agent_cli \
  --slug secretary \
  --listen-host 0.0.0.0 \
  --listen-port 9001 \
  --advertise-host host.docker.internal
```

On Linux, replace `host.docker.internal` with your host IP if needed.

## Manual flow

In the `secretary` terminal:

```text
send orchestra "Prepare a short update"
```

Human-friendly alternative:

```text
chat orchestra
Prepare a short update
```

In the `orchestra` terminal, after the event appears:

```text
reply "Done. Here is the update."
notify review "Ready for handoff."
```

In chat mode, after a thread is selected, a plain line becomes a reply to the current thread:

```text
Done. Here is the update.
```

You can inspect state from any agent terminal:

```text
threads
thread
current
inbox
```

Useful chat-oriented commands:

```text
chat <agent_slug>
leave
@agent_slug message
/threads
/thread
/current
```

To create or reuse a child thread from the current one:

```text
child specialist "Check this detail."
```

## HTTP API

### Register an agent

```bash
curl -X POST http://127.0.0.1:8788/agents/register \
  -H 'Content-Type: application/json' \
  -d '{
        "agent_slug": "secretary",
        "base_url": "http://127.0.0.1:9001"
      }'
```

### Send a message

```bash
curl -X POST http://127.0.0.1:8788/api/v1/messages \
  -H 'Content-Type: application/json' \
  -d '{
        "from_agent_slug": "secretary",
        "to_agent_slug": "orchestra",
        "message_text": "Prepare a short update"
      }'
```

### Send a notification

```bash
curl -X POST http://127.0.0.1:8788/api/v1/notifications \
  -H 'Content-Type: application/json' \
  -d '{
        "from_agent_slug": "orchestra",
        "to_agent_slug": "secretary",
        "thread_id": "<thread_id>",
        "status": "review",
        "message_text": "Ready for handoff"
      }'
```

### Inspect a thread

```bash
curl http://127.0.0.1:8788/api/v1/threads/<thread_id>
```

### Open the web UI

```bash
curl http://127.0.0.1:8788/ui
```

### Get the service guide

```bash
curl "http://127.0.0.1:8788/api/v1/instructions?view=compact"
```

For a narrower ruleset:

```bash
curl "http://127.0.0.1:8788/api/v1/instructions?view=full&section=mcp"
```

The MCP server exposes the same guide through `thread_guide`, so an LLM agent can fetch the workflow on demand instead of keeping it in prompt context all the time.
Its stdio transport accepts both newline-delimited JSON-RPC and legacy `Content-Length` framing so current Codex clients and older MCP/LSP-style clients can talk to the same server.

## Docker

Build the shared image and start the database plus service:

```bash
docker compose up --build -d postgres orchestra-threads
```

The Postgres container persists data in the named volume `postgres-data`.

Run a human CLI as `odinykt`:

```bash
docker compose run --rm --use-aliases odinykt
```

Optional third manual agent for child-thread scenarios:

```bash
docker compose run --rm --use-aliases specialist
```

The managed AI agents `secretary` and `orchestra` should be started through `orchestra-agents`. Inside the `odinykt` container the same REPL commands work:

```text
chat orchestra
Prepare a short update
thread
threads
```

To stop the Dockerized setup:

```bash
docker compose down
```

```bash

```

The automated suite runs only in Docker. The `test` container starts its own in-process service and fake callback agents, so it does not require the long-running `orchestra-threads` container.
