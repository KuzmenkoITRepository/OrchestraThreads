# OrchestraThreads MVP Draft

## Goal

`OrchestraThreads` is a standalone inter-agent thread service.

For this MVP:

- the service is a separate HTTP process or container;
- work identity lives in `thread_id`;
- delivery is HTTP callback based;
- persistence is Postgres;
- agents are manual CLI processes instead of LLM runtimes.

## Agent Interface

On startup an agent registers itself:

- `POST /agents/register`
- fields:
  - `agent_slug`
  - `base_url` or explicit `event_callback_url` and `stop_callback_url`
  - optional `display_name`
  - optional `metadata`

The CLI agent also sends `POST /agents/heartbeat` periodically.

Callbacks from `OrchestraThreads` to the agent:

- `POST /event`
- `POST /stop`

`/event` receives a batch of events:

- `message`
- `notification(in_progress|review)`
- `inactive`

`/stop` is control-plane only and is triggered when:

- a thread is `closed`;
- a parent thread becomes terminal and child threads are cascaded closed.

## Thread Model

Thread is:

- pairwise;
- durable;
- bidirectional;
- owned by the agent that created it with the first `message`;
- optionally a child of another thread.

Fields:

- `thread_id`
- `root_thread_id`
- `parent_thread_id`
- `owner_agent_slug`
- `participant_a_agent_slug`
- `participant_b_agent_slug`
- `status`
- activity timestamps

## Routing Rules

`POST /api/v1/messages`:

- with `thread_id`: continue that exact thread;
- with `parent_thread_id`: create or reuse a child thread under that parent;
- with neither: create or reuse the active root thread for the pair.

Child routing is the manual equivalent of `orchestra -> specialist` while processing another thread.

## Status Rules

Thread statuses:

- `open`
- `in_progress`
- `review`
- `done`
- `closed`

Permissions:

- owner can send `in_progress`, `done`, `closed`;
- peer can send `in_progress`, `review`.

Terminal rules:

- `done` and `closed` are terminal;
- terminal threads reject new messages or notifications;
- `closed` triggers peer `/stop`;
- terminal parent threads cascade-close live child threads.

## Event Rules

Events persisted in the service:

- `message`
- `notification`
- `inactive`

`message`:

- always delivered to peer via `/event`;
- requires a substantive response;
- updates `last_message_sender_agent_slug`;
- resets inactivity.

`notification`:

- `in_progress` and `review` are delivered via `/event`;
- `done` and `closed` are control-plane only;
- only `in_progress` and `review` reset inactivity.

`inactive`:

- generated by the service after timeout;
- delivered to the last agent that sent a `message`;
- does not itself reset inactivity.

## Delivery Rules

Delivery is `at-least-once`.

The service keeps pending events and retries with backoff until the target agent is online and accepts the callback.

This MVP deliberately prefers observability and manual testing over strict distributed guarantees.
