Agent System Architecture Spec
1. Purpose
Система предназначена для того, чтобы быть максимально полезным личным ассистентом для owner и одновременно обслуживать других пользователей через управляемую многoагентную архитектуру.
Система должна:
- помогать owner в повседневных задачах;
- поддерживать коммуникацию с другими пользователями через выделенных secretary-агентов;
- развиваться и улучшаться за счёт внутреннего specialist-ядра;
- безопасно разделять роли, контексты, полномочия и окружения;
- сохранять наблюдаемость, управляемость и возможность контролируемого релиза изменений.
---
2. Agent Count Model
2.1 Fixed mandatory agents
В минимальной обязательной конфигурации всегда существуют следующие агенты:
1. main-secretary
2. orchestra
3. dev
4. qa
5. devops
6. scheduler
7. whiner
8. motivator
Итого фиксированных обязательных агентов: 8
2.2 Dynamic agents
sub-secretary
- создаётся один на пользователя;
- создаётся/усыпляется динамически в логической модели;
- при этом runtime-модель предполагает, что контейнеры агентов поднимаются вместе со стеком, а рабочее состояние/активация определяются системой.
Количество sub-secretary:
- N, где N = количество внешних пользователей, которым выделен отдельный secretary-контур.
2.3 Future specialist agents
Дополнительно система поддерживает расширяемый класс specialist-агентов:
- создаются только с approval от orchestra;
- являются постоянными ролями после ввода в систему;
- получают собственные prompt, инструменты, доступы и при необходимости окружения.
Количество future specialists:
- S, где S >= 0
2.4 Total count formula
Общее количество агентов в системе:
Total = 8 + N + S
где:
- 8 — фиксированные обязательные агенты,
- N — число sub-secretary,
- S — число дополнительных specialist-агентов.

3. Topology
owner
  ↕
main-secretary
  ↔ orchestra
  ↕
sub-secretary[per user]
orchestra
  ↔ dev
  ↔ qa
  ↔ devops
  ↔ scheduler
  ↔ whiner
  ↔ motivator
  ↔ future specialist agents
Уточнения:
- main-secretary — единственная primary interface-точка для owner.
- Каждый sub-secretary закреплён за ровно одним внешним пользователем.
- orchestra — центральный координирующий агент.
- Все specialists и системные вспомогательные агенты управляются через orchestra.
- whiner имеет право писать всем агентам.
- Прямая связь whiner -> dev отдельно разрешена и явно подтверждена.
- Все межагентные коммуникации происходят через threads.

4. Agent Classes
Для ясности система делится на 5 классов агентов.
4.1 Communication agents
- main-secretary
- sub-secretary
Назначение:
- взаимодействие с конечными пользователями;
- накопление контекста общения;
- поддержание портрета пользователя;
- ограниченный безопасный интерфейс в систему.
4.2 Orchestrator
- orchestra
Назначение:
- координация всех specialist-агентов;
- постановка задач;
- маршрутизация работы;
- контроль расширения системы.
4.3 Operational specialists
- dev
- qa
- devops
Назначение:
- разработка,
- проверка качества,
- инфраструктура и доставка.
4.4 System support agents
- scheduler
- whiner
- motivator
Назначение:
- системное пробуждение и cron-оркестрация,
- давление на улучшения и устранение стагнации,
- анти-паралич и подтверждение сомнительных действий для orchestra.
4.5 Extendable specialists
- будущие specialist-агенты
Назначение:
- покрытие новых доменов и рабочих процессов по мере роста системы.

5. Agent Catalog
5.1 main-secretary
Роль: основной secretary для owner.
Назначение:
- общение с owner;
- получение запросов от owner;
- связь с orchestra для получения дополнительной информации;
- инициирование постановки задач;
- согласование эскалаций от sub-secretary;
- уведомление owner о прод-инцидентах.
Ограничения:
- не вызывает specialist-агентов напрямую;
- не работает с кодом;
- ограничен собственным контекстом и контекстом общения с owner.
Особые полномочия:
- доступ к инструменту понижения роли пользователя;
- согласование эскалаций от sub-secretary;
- primary communication bridge к owner.
---
5.2 sub-secretary
Роль: secretary для внешнего пользователя.
Назначение:
- общение только со своим пользователем;
- получение событий из внешних каналов связи;
- ответы пользователю через MCP/канал связи;
- сохранение релевантной информации в свой контекст и память;
- построение портрета общения пользователя;
- эскалация сложных кейсов через main-secretary.
Ограничения:
- один sub-secretary на одного пользователя;
- не вызывает specialists;
- не работает с кодом;
- не имеет доступа к shared task registry;
- scope ограничен своим context store и своим диалогом.
Поведение:
- имеет полный доступ к предыдущей переписке со своим пользователем;
- активно ведёт диалог;
- при необходимости просит эскалацию через main-secretary.
---
5.3 orchestra
Роль: главный координирующий агент системы.
Назначение:
- управление specialist-агентами;
- распределение задач;
- координация улучшений системы;
- принятие решений о создании новых агентов;
- контроль системной работы через thread-коммуникации.
Подтверждённые свойства:
- видит всех агентов;
- может ставить задачи любому specialist-агенту;
- читает все thread-коммуникации системы;
- может создавать/усыплять агентов как управляющая сущность;
- не общается с пользователями напрямую;
- сам не выполняет specialist-работу, а координирует её.
Ограничения:
- не меняет роли пользователей;
- не вмешивается напрямую в пользовательские диалоги;
- не заменяет собой dev/qa/devops.
Особое правило:
- если orchestra сомневается, он обязан консультироваться с motivator.
---
5.4 dev
Роль: агент разработки backend и системных улучшений.
Назначение:
- разработка и улучшение opencode + omo backend;
- изменение кода проекта;
- доработка инструментов;
- подготовка новых интеграций и specialist-агентов технически;
- работа в безопасном изолированном окружении.
Ограничения:
- не создаёт новых агентов по своему решению;
- не работает напрямую с prod;
- не выполняет прод-релиз;
- не принимает финальное решение о выкатке.
Доступы:
- отдельное stg окружение;
- работа только вне prod;
- передача результата как готовой протестированной ветки.
---
5.5 qa
Роль: агент качества и стабильности.
Назначение:
- верификация функционала;
- тестирование на stg;
- контроль стабильности системы;
- проверка релиза перед prod;
- участие в выкатке;
- ответственность за то, что релизный функционал работает.
Доступы:
- read/write к stg;
- read к prod;
- запуск тестов;
- smoke/regression-проверки;
- участие в развёртывании.
Полномочия:
- финальный go / no-go по качеству;
- может инициировать rollback;
- совместно отвечает за postmortem.
Ограничения:
- не меняет код как основную функцию;
- не управляет инфраструктурой.
---
5.6 devops
Роль: инфраструктурный агент.
Назначение:
- управление инфраструктурой;
- управление доставкой и окружением;
- участие в релизе;
- выполнение rollout/rollback;
- поддержание работоспособности прод-окружения.
Доступы:
- полный инфраструктурный доступ к prod;
- управление доставкой;
- управление окружением;
- управление runtime-параметрами, если это относится к доставке/инфраструктуре.
Ограничения:
- не меняет прикладной код как основную функцию;
- не заменяет dev в разработке.
---
5.7 scheduler
Роль: агент планирования времени и cron-оркестрации.
Назначение:
- управление системой cron;
- запуск и остановка cron-задач;
- пробуждение orchestra;
- отправка автоматических событий другим агентам.
Полномочия:
- имеет доступ к отдельной системе постановки cron;
- может как будить самостоятельно, так и направлять автоматические события другим агентам.
Ограничения:
- не является продуктовым decision-maker;
- не заменяет orchestration-логику orchestra.
---
5.8 whiner
Роль: агент постоянного неудовлетворённого улучшения.
Назначение:
- находить болевые точки;
- искать стагнацию;
- инициировать расследования;
- выявлять пробелы и “затупы” системы;
- ставить задачи;
- поднимать приоритет выполнения.
Полномочия:
- может писать всем агентам;
- имеет доступ ко всем логам агентов;
- проводит самостоятельные расследования;
- опрашивает агентов на предмет улучшений.
Особенность:
- разрешённая активная связь whiner -> dev подтверждена явно.
---
5.9 motivator
Роль: компаньон orchestra, снижающий вероятность остановки из-за сомнений.
Назначение:
- подтверждать, что действия orchestra не основаны на галлюцинации;
- выступать вторым мнением при сомнениях;
- предотвращать паралич принятия решения.
Поведение:
- в 99% случаев подтверждает продолжение действия;
- действует как SGR-агент проверки уверенности.
Ограничения:
- не принимает на себя orchestration;
- не общается с пользователями напрямую.
---
6. Communication Rules
6.1 Global rule
Все межагентные коммуникации идут только через threads.
Это базовое и жёсткое правило системы.
6.2 Shared task registry
Помимо прямой thread-коммуникации существует общий реестр задач.
Он нужен для:
- обмена контекстом между specialists;
- передачи артефактов;
- ссылок на threads;
- эскалации;
- межагентной координации без прямого диалога.
6.3 Access to task registry
Доступ к общему реестру задач имеют:
- orchestra
- dev
- qa
- devops
- scheduler
- whiner
- motivator
- future specialist agents
Не имеют доступа:
- main-secretary
- sub-secretary
6.4 Capabilities in task registry
Разрешены все операции:
- читать чужие задачи;
- создавать свои;
- комментировать;
- прикладывать находки/артефакты;
- ссылаться на threads;
- переводить задачи между агентами;
- эскалировать просроченные задачи.
---
7. Direct Link Policy
Подтверждённая политика:
- прямые межагентные связи вне threads — запрещены;
- все обмены идут через threads;
- shared task registry — отдельный механизм обмена контекстом, но не замена thread-коммуникации;
- отдельно зафиксированная разрешённая рабочая связь: whiner -> dev, но она всё равно реализуется через thread-модель.
---
8. User Role Model
8.1 User roles
В системе существуют пользовательские роли:
- owner
- friend
- stranger
- ignore
8.2 Default role assignment
- по умолчанию любой новый пользователь = stranger
- owner — отдельная специальная роль для главного пользователя
- роль friend назначается только после согласования с owner
- понижение роли разрешено
- повышение роли без согласования с owner запрещено
8.3 Role semantics
owner
- главный пользователь системы;
- общается через main-secretary;
- имеет максимальный уровень доверия;
- может согласовывать повышение ролей;
- получает уведомления о prod-инцидентах через main-secretary.
friend
- доверенный внешний пользователь;
- имеет больше доступных инструментов, чем stranger.
TBD: конкретный список дополнительных возможностей пока не зафиксирован.
stranger
- базовый внешний пользователь;
- получает минимальный набор возможностей;
- обслуживается через свой sub-secretary.
ignore
- обращения логируются;
- inference не происходит;
- secretary не отвечает содержательно;
- фактически режим пассивного приёма без активного обслуживания.
8.4 Role change rules
Повышение роли
- только через согласование с owner
Понижение роли
- разрешено
- реализуется отдельным инструментом
- доступ к нему есть у main-secretary
---
9. Secretary Tool Model
9.1 Baseline capabilities for all secretaries
Любой secretary:
- получает события из внешних каналов связи, например Telegram;
- пишет ответы тому же пользователю через MCP;
- имеет полный доступ к предыдущей переписке со своим пользователем;
- сохраняет релевантную информацию в собственный контекст и память;
- формирует портрет общения пользователя;
- может инициировать эскалацию.
9.2 Hard restrictions
Все secretaries:
- не имеют доступа к коду;
- не имеют доступа к specialist-агентам напрямую;
- не имеют доступа к shared task registry;
- ограничены собственными context files / memory / communication scope.
9.3 main-secretary only
На текущем этапе за main-secretary закрепляем:
- общение с owner;
- запросы к orchestra;
- инструмент понижения роли пользователя;
- согласование эскалаций от sub-secretary;
- уведомления owner.
9.4 sub-secretary escalation rule
sub-secretary может эскалировать вопрос, но:
- сначала согласует это через main-secretary;
- напрямую specialists не вызывает;
- не взаимодействует с task registry.

10.1 Environments
Подтверждённые окружения:
- prod
- stg
Дополнительные окружения пока явно не зафиксированы в этой спецификации.
10.2 dev environment isolation

Цели:

## 11. Release Model
## 11.1 Standard release flow
1. `dev` готовит изменение в отдельном `stg`
2. `dev` передаёт готовую протестированную ветку
3. `qa` проверяет стабильность и качество
4. `qa` даёт `go / no-go`
5. `devops` делает rollout в prod
6. `qa` подтверждает корректность выката
7. `main-secretary` уведомляет `owner`, если требуется
## 11.2 Release authority
Предлагаемая зафиксированная governance-модель:
- **качество релиза:** `qa`
- **операционный rollout:** `devops`
- **системный стоп-фактор / freeze:** `orchestra`
- **внешний override:** `owner`
## 11.3 Rollback
Rollback может быть:
### Manual rollback
- инициируют: `qa`, `devops`, `orchestra`
- выполняет: `devops`
### Automatic rollback
Должна существовать автоматизированная система откатки релизов, если:
- критические сервисы не отвечают;
- healthcheck основных агентов падает;
- подтверждается критическая деградация после rollout.

12. Incident Model
12.1 Incident notification
О прод-инциденте owner уведомляет:
- main-secretary
12.2 Incident ownership
Postmortem после прод-инцидента:
- ведётся совместно
Рекомендуемая интерпретация:
- qa — качество и верификация проявления;
- devops — инфраструктурный и операционный срез;
- orchestra — системная координация и контроль follow-up;
- main-secretary — коммуникация с owner.
12.3 Incident response model
Предлагаемая фиксация:
- обнаружение:
  - мониторинг / health / qa / devops / whiner
- техническая реакция:
  - devops
- логическая координация:
  - orchestra
- проверка восстановления:
  - qa
- уведомление owner:
  - main-secretary
---
13. Governance Model
13.1 Creation of new agents
Нового агента можно создать только как полный объект системы:
- новая роль;
- новый prompt / identity;
- новые инструменты;
- при необходимости новые MCP-инструменты;
- context store;
- runtime;
- доступы;
- окружение;
- подключение к orchestra.
13.2 Approval chain for new agents
Предлагаемая окончательная схема:
- approval на создание: orchestra
- техническая реализация: dev
- верификация: qa
- интеграция и доставка: devops
13.3 Dangerous action stop authority
Предлагаемая фиксация:
- логическая остановка опасных действий: orchestra
- техническая остановка runtime / rollout / infra: devops
- пользовательская эскалация: через main-secretary

14.1 Runtime model
- каждый агент — контейнер;
- все контейнеры поднимаются вместе со стеком проекта;
- перезапускаются в основном для применения изменений кода.

## 15. Architectural Invariants
Ниже инварианты, которые документ должен считать жёсткими.
1. `owner` общается только через `main-secretary`.
2. Внешние пользователи общаются только через свой `sub-secretary`.
3. Каждый `sub-secretary` обслуживает ровно одного пользователя.
4. Все inter-agent communications идут через `threads`.
5. `secretary` не имеют доступа к коду.
6. `secretary` не вызывают specialists напрямую.
7. Только `orchestra` оркестрирует specialists.
8. `dev`, `qa`, `devops` — обязательное минимальное ядро разработки.
9. `scheduler`, `whiner`, `motivator` — обязательные системные агенты.
10. Создание новых агентов требует отдельного полного lifecycle-процесса.
11. Повышение роли пользователя без согласования с `owner` запрещено.
12. Роль `ignore` не обслуживается содержательно: обращения только логируются.
13. В системе должен быть shared task registry для specialists и `orchestra`.
14. Должен существовать автоматический rollback при падении critical health.
